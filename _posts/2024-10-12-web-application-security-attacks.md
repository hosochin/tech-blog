# 主な攻撃手法とその対策について

| 攻撃手法 | 脆弱性が生まれる原因 | 対策 |
|---|---|---|
| XSS（クロスサイト・スクリプティング） | ・HTMLのエスケープが適切でない<br>・出力時のエスケープ不足 | ・要素内容については「<」「&」をエスケープする<br>・属性値についてはダブルクォートで囲って、「<」「"」「&」をエスケープする<br>・HTTPレスポンスに文字エンコーディングを明示する<br>・入力値の検証、サニタイズ<br>・出力時のエスケープ処理<br>・cookieにHttpOnly属性を付与<br>・（TRACE メソッドの無効化）<br>・（X-XSS-Protection レスポンスヘッダの使用） |
| SQLインジェクション | ・動的なSQLクエリの作成<br>・入力値の検証不足<br>・エラーメッセージの露出 | ・プレースホルダによりSQL文を組み立てる<br>・可能であれば静的プレースホルダを使う<br>・詳細なエラーメッセージの抑止<br>・入力値の検証<br>・データベースの権限設定 |
| CSRF（クロスサイト・リクエストフォージェリ） | ・form要素のaction属性にはどのドメインのURLでも指定できる<br>・cookieに保管されたセッションIDは対象サイトに自動的に送信される | ・秘密情報（CSRFトークン）の埋め込み<br>・パスワード再入力<br>・Refererのチェック |
| クリックジャッキング | ・HTMLの仕様の悪用 | ・X-Frame-Optionsヘッダーの使用<br>・「重要な処理」の実行後には通知メールを送信する |
| セッションハイジャック | ・主要なWebアプリケーション開発ツールはセッション管理機構を備えている<br>・安全なセッションID生成処理を開発することは技術的難易度が高い | ・Webアプリケーション開発ツールなどが備えるセッション管理機構を利用する（極力自作しない）<br>・cookieにセッションIDを保存する<br>・認証生工事にセッションIDを変更する<br>・認証前にはセッション変数に秘密情報を保持しない |
| セッションIDの固定化攻撃 | ・セッションIDを外部から強制できる仕組みになっている | ・セッションハイジャックの対策と同様 |
| オープンリダイレクト脆弱性を悪用した攻撃 | ・リダイレクト先のURLを外部から指定できる<br>・リダイレクト先のドメイン名のチェックがない | ・リダイレクト先のURLを固定にする<br>・リダイレクト先のURLを直接指定せず番号指定にする<br>・リダイレクト先のドメイン名をチェックする |
| HTTPヘッダ・インジェクション | ・HTTPヘッダの1行に1つのヘッダを定義でき、改行で区切られている性質を悪用する | ・外部からのパラメータをHTTPレスポンスヘッダとして出力しない<br>・リダイレクトやcookie生成を専用API（ライブラリ）にまかせる<br>・ヘッダ生成するパラメータの改行文字をチェックする |
| cookieのセキュア属性不備を利用した攻撃 | ・開発者がcookieのセキュア属性について知らない<br>・セキュア属性をつけるとアプリケーションが動かなくなる（HTTPとHTTPSが混在するWebアプリケーションなど） | ・cookieにセキュア属性を付与する |
| メールヘッダ・インジェクション | ・ヘッダの各フィールドは改行で区切られているという、メールヘッダの仕様を悪用 | ・メール送信には専用のAPIやライブラリを使用する<br>・外部からのパラメータをメールヘッダに含ませないようにする<br>・外部からのパラメータには改行を含まないようにメール送信時にチェックする<br>・メールアドレスのチェック<br>・件名のチェック |
| ディレクトリ・トラバーサル | ・ファイル名を外部から指定可能<br>・ファイル名として、絶対パスや相対パスの形で異なるディレクトリを指定できる<br>・組み立てたファイル名に対するアクセスの可否をチェックできていない | ・外部からファイル名を指定できる仕様を避ける<br>・ファイル名にディレクトリ名が含まれないようにする<br>・ファイル名を英数字に限定する |
| OSコマンド・インジェクション | ・シェルを呼び出す機能のある関数を利用している<br>・シェル呼び出しの機能のある関数にパラメータを渡している<br>・パラメータ内に含まれるシェルのメタ文字をエスケープ漏れ | ・設計フェーズで対策方針を決定する<br>・OSコマンド呼び出しを使わない実装方法を選択する<br>・シェル呼び出し機能のある関数の利用を避ける<br>・外部から入力された文字列をコマンドラインのパラメータに渡さない<br>・OSコマンドに渡すパラメータのエスケープ処理<br>・アプリケーションの稼働するユーザ権限を最小限にする |
| ファイルアップロード機能を悪用した攻撃 | ・アップロードしたファイルが公開ディレクトリに保存される<br>・サーバースクリプトを示す拡張子を指定できる | ・公開ディレクトリに保存しない |

# まとめ

Webアプリケーション開発において、セキュリティ対策は非常に重要です。徳丸本で紹介されている攻撃手法とその対策を理解し、実装時に適切な対策を講じることで、安全なWebアプリケーションを構築できます。特に入力値の検証、出力時のエスケープ処理、適切な認証・認可の実装は基本中の基本として押さえておきたいところです。
